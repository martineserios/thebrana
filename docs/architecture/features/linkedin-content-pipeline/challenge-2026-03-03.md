## Critical (3) — must fix before build

### C1: sessions.md path and schema are misunderstood

The feature design says sessions.md is the "primary fuel source" but never specifies its actual path. The file lives at `~/.claude/projects/.../memory/sessions.md` — outside the repo, auto-generated by `session-end.sh`. Real entries are frequently sparse (0-event venture-start sessions) and the "Files" field contains corrupted fragments. There is no formal schema contract — it is an implementation detail of session-end.sh, not an API.

**Fix:** Formalize the sessions.md contract before building t-166: document the actual absolute path, define the schema, decide whether claude-flow `memory_search` (richer structured JSON) is a better primary source.

### C2: 60% carousels with no carousel tool is a v1 blocker

At 3 posts/week with 60% carousels = ~2 carousels/week. `/brana:export-pdf` produces plain PDFs via mdpdf — not designed slides. Each carousel requires 30-60 min of manual Canva/Figma work. The 2-hour Sunday batch allocates 0 minutes for visual design.

**Fix:** Either (a) drop carousel target to 20-30% for v1, (b) create pre-built Canva templates that accept markdown text, or (c) accept plain PDFs initially with a v2 visual upgrade.

### C3: Scoring formula is untestable and creates false confidence

`novelty×3, engagement×2, balance×2, recency×1` — recency is meaningless (all candidates are from last 7 days), engagement potential is a single heuristic, pillar balance is a constraint not a signal.

**Fix:** Replace with: filter (pillar balance + novelty via published.md) → present grouped by pillar → human picks 3.

## Warnings (5) — significant risks needing mitigation

### W1: published.md as deduplication state is a manual SPOF

If the user forgets to update published.md, the skill suggests repeated topics. Trust erodes, tool abandoned.

**Mitigation:** Make published.md update part of `/content-draft` workflow — when user confirms a draft was published, the skill appends automatically.

### W2: Sunday batch is a single point of failure with no buffer

One missed Sunday = zero posts that week. Missing 2 weeks in months 1-3 undermines the 90-day strategy.

**Mitigation:** Require a 3-post buffer maintained at all times. t-165 (first 6 posts) starts this.

### W3: Creator engagement risks looking spammy

Cold-tagging 52 creators who don't know the author. Studying someone's content is not a relationship.

**Mitigation:** 4-6 weeks of meaningful comments BEFORE any tagging. Reduce tier 1 to 2-3 creators initially.

### W4: Consulting as downstream outcome is an assumption, not a strategy

Build-in-Public (40%) and Contrarian (20%) signal thought, not commercial capability. Case Studies (10%) is the only "hire me" pillar — smallest allocation.

**Mitigation:** Either drop consulting as v1 goal or increase Case Studies to 20%.

### W5: "Golden hour" conflicts with batch scheduling

First 60-90 min after posting determine reach — requires being online. Buffer schedules at 8-9 AM. If the author can't respond then, reach is negated.

**Mitigation:** Schedule posts for times when the author IS available, not a fixed window.

## Suggestions (4)

### S1: Publish manually for 4 weeks before building the skill

Both Gemini and the challenger independently recommended this. Manual posting validates pillar weights, reveals the real editing burden, and produces sessions.md data. Build /content-draft after 12+ manual posts — design from experience, not theory.

### S2: Reclassify from `experiments` to `roadmap`

ph-005 is tagged stream: experiments. A 6-month pipeline with 8 tasks and 3 milestones is a roadmap commitment.

### S3: Add a voice/tone guide

No documentation for what makes the author's writing "authentic." Write 3-5 example posts in natural voice, use as templates.

### S4: Add fallback for quiet weeks

When brana has no activity, the 9 named frameworks and 580+ knowledge entries are the fallback. Document this as "evergreen mode" in the skill design.
