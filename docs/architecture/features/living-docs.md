# Feature: Living Documentation System

**Date:** 2026-03-15
**Status:** specifying
**Task:** t-476

## Problem

Documentation drifts from implementation. The build CLOSE step generates feature docs, but:
1. No composable building block — doc generation is embedded in CLOSE, not reusable
2. No spec-graph routing — can't determine which existing docs need updating when code changes
3. No philosophy doc — users don't understand the "why" behind the system
4. Shared docs (commands/index.md, workflows/) require manual awareness to update

## Decision Record (frozen 2026-03-15)

> Do not modify after acceptance.

**Context:** t-069 investigation mapped the full doc architecture. Challenger review refined the approach: auto-generated routing via spec_graph.py extension, post-merge follow-up instead of inline CLOSE blocker, diff preview for shared doc updates.

**Decision:** Create `/brana:docs` as a standalone composable skill with subcommands (guide, tech, overview, all). Extend spec-graph.json with `guide_files`, `arch_files`, `ref_files` fields for routing. CLOSE invokes `/brana:docs all` as a post-build step. Philosophy doc seeds the "why".

**Consequences:** Any skill/command can invoke `/brana:docs` (CLOSE, reconcile, maintain-specs, manual). Spec-graph routing replaces convention-based matching. New skills auto-add spec-graph nodes.

## Constraints

- Composable: `/brana:docs` must work standalone AND as a building block for other skills
- Spec-graph routing auto-generated by extending spec_graph.py, not manual mapping
- Auto for new docs, diff preview for shared doc updates
- Philosophy doc is a seed — grows over time, not a one-shot write

## Scope (v1)

1. `/brana:docs` skill with subcommands: `guide`, `tech`, `overview`, `all`
2. Extend spec-graph.json schema with doc-type routing fields
3. Update spec_graph.py to populate new fields
4. Philosophy doc at `docs/guide/philosophy.md`
5. CLOSE step calls `/brana:docs all` (replaces inline doc generation)

## Research

- t-069 investigation: mapped 173 nodes, 534 edges in spec-graph, 3 doc layers (guide/architecture/reference)
- Challenger finding: convention-based matching breaks (build touches multiple docs) — use spec-graph mapping
- Challenger finding: standalone skill confirmed as composable building block pattern (same as brana backlog CLI)
- Existing templates: tech-doc.md (48 lines), user-guide.md (52 lines) already in build/templates/

## Design

### Skill structure
```
system/skills/docs/
├── SKILL.md          ← skill definition with subcommands
```

### Subcommands
- `guide` — generate/update user guide for a feature
- `tech` — generate/update tech doc for a feature
- `overview` — generate/update philosophy.md with latest patterns
- `all` — run guide + tech + overview (used by CLOSE)

### Spec-graph extension
New fields per node:
```json
{
  "guide_files": ["docs/guide/features/X.md", "docs/guide/commands/index.md"],
  "arch_files": ["docs/architecture/features/X.md"],
  "ref_files": ["docs/reference/skills.md"]
}
```

### CLOSE integration
Replace inline doc generation in build CLOSE with:
```
Skill(skill="brana:docs", args="all {task-id}")
```
