# ADR-066: `active_epic`/`active_initiative` Resolve Project-Local Only

- **Status:** Accepted
- **Date:** 2026-07-22
- **Evidence:** t-2158 (commit `44e584fd`, 2026-06-20) already implemented this in code; t-2281 (this task) formalizes it as policy and closes the residual gaps found during audit
- **Related:** t-2155 (`set-active` writes project-local), t-1883 (sync-state.sh push-side contamination guard), ADR-065 (epic-as-hierarchy-top), [pattern: active-epic-two-copies](../../../../.claude/memory) (deployed vs repo config copies)

## Context

`active_epic` and `active_initiative` boost focus/ranking for one project's work. Each lives in exactly one repo — an epic never spans projects. Before t-2158, `brana backlog focus`/`add` read `active_epic` from whichever `tasks-config.json` resolved (project-local if present, else the global `~/.claude/tasks-config.json`) with no distinction between "no local value" and "inherit the global value." A project with no local config silently inherited whatever epic another project's session had last set globally — observed directly on thebrana, which had no project-local config and was inheriting `identity-commerce-rebuild`, a proyecto_anita value.

t-2158 fixed the read path in `brana_core::util::load_tasks_config()`: a project-local config, if present, is authoritative; falling back to global strips `active_epic`/`active_initiative` (`PROJECT_SCOPED_CONFIG_KEYS`) before returning. t-2155 fixed the CLI write path: `cmd_set_active` writes to the resolved config path, which is project-local whenever a project root is determinable.

**Challenger review (2026-07-22) found this fix is narrower than first claimed.** Re-auditing during t-2281 surfaced three gaps not covered by t-2158/t-2155, one of them live and high-traffic:

1. **Two skill procedures bypass the resolver entirely, in prose.** `system/skills/backlog/phases/plan.md` (step 3a) and `system/skills/backlog/phases/done-and-add.md` (step 3a) both instruct: read `active_epic` from `~/.claude/tasks-config.json` — the literal global path, hardcoded, with no mention of the scoped resolver — then auto-assign it to new tasks. These are the instructions an LLM session follows for `/brana:backlog add` and `/brana:backlog plan`, the two highest-traffic task-creation entry points. A project with no local config following these steps today reproduces the original cross-project bleed, in the exact scenario this ADR claims is now impossible. This also means the Non-Actions claim below (`add`'s auto-inheritance "doesn't exist yet") was wrong — it exists as a documented procedure, just not as Rust code.
2. **`cmd_set_active` still falls back to the global file** when no project root is determinable (no git repo, and no pre-existing local `.claude/` at CWD) — confirmed via `find_tasks_config_from()`/`find_tasks_file_from()`: with no `common_root` and no `toplevel`, and CWD's `.claude/` absent, resolution returns `None` and `tasks_config_path()` falls back to `global_tasks_config_path()`. Narrow (requires running `set-active` outside any git repo with no local config already seeded), but it means "never writes to global directly" was not quite true.
3. **`system/scripts/sync-state.sh`** operates at a different layer — it's thebrana's own deploy sync between its tracked `system/state/tasks-config.json` and the global `~/.claude/tasks-config.json` cache. `cmd_push` (cache → repo) has a contamination guard (t-1883): if the global cache's `active_epic` diverges from what thebrana's repo copy had before the push, it reverts to the repo's prior value and warns. `cmd_pull` (repo → cache, what `bootstrap.sh` runs) had **no equivalent guard** — pulling can silently overwrite the global file's `active_epic` with thebrana's own value, clobbering it for any project relying solely on the global copy.
4. **Orphaned global state**: the global `~/.claude/tasks-config.json` carried `active_epic: "identity-commerce-rebuild"` — a proyecto_anita epic slug, but *not* proyecto_anita's current epic (`~/enter_thebrana/ventures/proyecto_anita/.claude/tasks-config.json` already has its own project-local `active_epic: "env-hardening"`, set independently since t-2155 shipped). The global value is confirmed dead: no project's resolution currently depends on it, since every project either has its own local config (post-t-2155/t-2158 world) or falls back to global-minus-project-scoped-keys (i.e., never sees it). It is leftover from before those fixes existed.

## Decision

**Policy, effective now:** `active_epic` and `active_initiative` are **project-scoped values with exactly one authoritative source: the resolving project's own `.claude/tasks-config.json`.** The global `~/.claude/tasks-config.json` is never a valid source of truth for either key — it may only hold non-project-scoped defaults (`theme`, `github_sync`). This formalizes what `PROJECT_SCOPED_CONFIG_KEYS` already enforces at the code level (t-2158) and what `set-active` already enforces at the write level (t-2155).

Concretely, closing the four gaps found above:

1. **Fix the skill procedures.** `plan.md` step 3a and `done-and-add.md` step 3a are rewritten to resolve `active_epic` via the scoped path — `brana backlog focus` output (which already calls `load_tasks_config()`) or an equivalent CLI/MCP call — never a raw read of `~/.claude/tasks-config.json`. This is the highest-priority item: it's the one gap with an active, silent-failure blast radius today.
2. **Hard-stop `cmd_set_active` when no project root is determinable**, instead of silently falling back to writing global. An explicit error ("no project root found — cannot resolve a scoped config path") is correct here: silently writing global is exactly the failure mode this ADR exists to eliminate, and there's no scenario where writing `active_epic` to the global file is the right outcome.
3. **`sync-state.sh cmd_pull` gets a guard mirroring the existing push-side one (t-1883), same algorithm, not a broader one:** capture the global cache's `active_epic`/`active_initiative` before the pull; after pulling, if thebrana's own repo value differs from what was there before AND a foreign (non-thebrana) value was present, warn and skip writing those two keys into the cache rather than clobber silently. This is a single before/after diff against thebrana's own prior state — deliberately not a comparison against every registered project (that design was considered and rejected, see the retired draft in the feature spec's Design section) because under this ADR's own target end-state (no project should have its value depend on the global key) that broader comparison degrades to "no registered project's local value ever matches," permanently blocking legitimate first-run cache seeding rather than converging to a no-op.
4. **One-time cleanup + reusable audit**: a migration script (`system/scripts/migrate/audit-orphaned-active-epic.py`) walks every project registered in `tasks-portfolio.json`, and for each one checks whether the global `~/.claude/tasks-config.json`'s `active_epic`/`active_initiative` value matches that project's *own* project-local config (if any). Any global value with no live project-local counterpart confirming it is reported as orphaned and cleared. This is written as a standing audit tool (rerunnable), not a one-shot patch, since new orphans can reappear until every project has migrated to local config and the skill-procedure + pull-side fixes fully close the write/read vectors.

## Consequences

- The two skill procedures (`plan.md`, `done-and-add.md`) get corrected — this closes the one gap with a live blast radius; until this ships, `/brana:backlog add`/`plan` remain a bleed vector on any project without a local config.
- `cmd_set_active` gains an explicit failure mode instead of a silent global-write fallback — a small, low-risk behavior change (only triggers outside any git repo with no local `.claude/` already present).
- `sync-state.sh` gains a small, testable guard mirroring existing t-1883 logic — same risk profile, same rollback story (revert-and-warn, never hard-fail), same single-repo-diff algorithm as the existing push guard (not a portfolio-wide comparison).
- The global `tasks-config.json` schema is unchanged; this is a policy/enforcement decision, not a schema migration.
- Future projects that never create a local `.claude/tasks-config.json` simply get no active-epic focus boost — this is correct behavior (a project that never opted in has no epic to boost), not a regression.

## Non-Actions

- Does **not** touch `active_epic`/`active_initiative` for projects that already have a correct project-local config (proyecto_anita is untouched — its `env-hardening` value is already correctly scoped).
- Does **not** change wave/epic-as-top schema work (ADR-065, t-2284) — orthogonal, this ADR is about *resolution scoping*, not the schema shape.
- Does **not** add a fourth parallel resolution path — `themes.rs::load_theme_name()` and `gh-sync.sh` independently re-read `tasks-config.json` but only for `theme`/`github_sync`, which are correctly non-project-scoped; no change needed there, but noted here so it doesn't get missed in a future audit.
