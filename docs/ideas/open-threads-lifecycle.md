# Open Threads Lifecycle

> Brainstormed 2026-06-22. Work in progress.

## Problem

Claude raises questions correctly — gaps, doubts, owed info, errors, blockers —
and scatters them across docs (idea docs, ADRs, feature specs) as prose
`## Open Questions` sections or inline blockers. When the answer / "the light"
later arrives (usually in chat), **nothing brings us back to the question.** We
rely on human/Claude tidiness to go remove the marker and reconcile the new info
into the doc — and tidiness does not scale. Result:

- **A. Stale docs** — the answer lands in chat, never written back. Doc stays wrong.
- **B. Context contamination** — the dead/answered question lingers and pollutes later reasoning.
- **C. No visibility** — no way to see what's actually open across the project.

Key reframe (2026-06-22): centralization (C) is NOT the core need. Questions
*should* live distributed, next to the decision they block. The core need is
**loop-closing** — an automated mechanism that finds open questions and resolves
them when the answer exists, instead of depending on discipline.

## Revised model — "Open Threads" (2026-06-22, from Finding #1 discussion)

> This supersedes the "central store as primary" framing below. The deliberation trail
> is kept, but the current model is this section.

**Concept: an "open thread"** — an open/unresolved item attached to a doc. Broader than a
question. `kind`: `question | blocker | unclear | awaiting-feedback | doubt`. Lifecycle:
**open → closed** (closed = answer reconciled into its targets). The noun is "open
thread"; the act of creating one is "flag" (`#thread` / `#open` capture).

**Three parts:**
1. **Doc-level markers = source of truth.** The open thread lives INLINE in the doc it
   belongs to. Everyday value: while reading/editing a doc you see what's open about it,
   in context. (This is why "extend `brana remind`" was rejected — remind has no
   doc-level presence; its items live in a central store, not inside the doc.)
2. **Central index is DERIVED, not maintained.** Generated by scanning docs for markers —
   gives the one-place view + reconcile sweep + cross-doc fan-out. Because it's derived,
   it can't drift from source (same reason `MEMORY.md` is regenerated, not edited). This
   resolves the derive-vs-declare fork AND the challenger's index-staleness HIGH risk.
3. **Reminder link is optional/opt-in.** A thread links to a `brana remind` entry ONLY
   if it needs a timed nudge — most don't. `remind` stays the orthogonal "ping me at a
   time / via a channel" tool (it has `--at`, `--channels`, snooze; NO doc targets). A
   thread is the primary object; a reminder is a thing it can point at.

**Why this beats both earlier options:** a pure central store loses doc-level everyday
use; extending `remind` loses it too AND couples to notifications. Doc-level markers +
derived index keeps in-context visibility, the one-place view, no new hand-maintained
store, and no remind overlap.

### Finding #2 resolution (effort) — SPLIT into two tasks

L, not M. Two backlog tasks: **Slice 1** = marker format + scanner/derived-index +
`/close` surface ("never lose an open thread" — ~80% of everyday value: doc-level
visibility + one-place view, manual resolution OK). **Slice 2** = reconcile-seeding +
dead-target check ("auto-propagate" — the hard, risky 20%), blocked_by Slice 1. Slice 1's
scanner is a baby `brana graph build` → seeds t-2223.

### Finding #3 resolution (capture) — forced checkpoint, not a mechanism

Capturing a thread is an irreducible judgment; reliability comes from forcing the judgment,
not from determinism. Three capture sources → candidate → triage → tracked:
1. **Cheap inline flag** (primary) — one-line marker, lower friction than a prose section.
2. **`AskUserQuestion` as capture signal** (welcomed) — each ask = a candidate thread.
   Answered in-session → auto-resolved, never promoted. **Only deferred/unanswered asks
   become tracked threads.** This filter is what keeps asktool-capture from flooding.
3. **Mandatory, unskippable `/close` checkpoint** (backstop) — best-effort detection of
   unflagged threads, but ALWAYS runs, so misses are bounded to one session.
Deterministic part = enumerating flagged markers (grep). Detection of unflagged threads
is best-effort by nature — stated honestly, not oversold.

### Finding #4 resolution (resolved-thread provenance) — DEFERRED to t-2228

Under the doc-level model the archive-growth worry mostly dissolves: the derived index
shows only OPEN markers, and git history already records each marker's add/reconcile/
remove. Open choice — (A) remove-on-close, git is the sole provenance vs (B) also append
to a queryable `threads-closed.jsonl`. **Deferred to t-2228, decide after usage feedback.**
Not needed for slice 1 (which doesn't close threads); a slice-2-time decision.

## Proposed solution (shape so far)

**Model: question-as-propagation-hub.** One canonical record per question, with
explicit fan-out edges to every artifact it blocks. (Refined 2026-06-22 — supersedes
an earlier "distributed markers" lean: a single question usually blocks MORE than one
doc/code file, and a marker living inside one doc can't express that fan-out.)

1. **Centralized question records.** Each record: `id`, question text, raised-by/date,
   `status`, and a list of **targets** — every doc/code location (file + optional
   anchor) that must be reconciled when the question is answered. This is a small
   graph; same shape as the existing `docs/spec-graph.json` (node + edges to
   `impl_files`). A question is a node with edges to the artifacts it blocks.
2. **Per-target resolution status ("partial light").** An incoming answer can be
   full or partial. Each target tracks its own state (pending → reconciled). Partial
   info advances the relevant targets; the question closes only when ALL targets are
   reconciled. Partial light must not falsely close a question.
3. **Layered automated trigger** (chosen): all three, operating on the central store —
   - **Push** — `/brana:log #answers <id> "..."` writes the answer and reconciles
     the targets you name, immediately.
   - **Pull** — `/brana:reconcile --scope questions` sweeps open records, detects
     answers in event-log / recent session, proposes write-backs per target.
   - **Close backstop** — `/close` lists questions that look answered this session
     and prompts to reconcile before context is lost.
4. **Human-in-the-loop confirmation gate.** Auto-detection (pull/close) *proposes*,
   the human *disposes*. A detected answer surfaces via `AskUserQuestion` confirming:
   (a) does this actually answer the question, (b) which targets does it cover —
   full or partial light, (c) apply now or defer. Only Push (explicit `#answers <id>`)
   may write back without the gate. This keeps the machine doing the tedious *finding*
   and the human owning the *judgment* — the split that fixes the tidiness problem
   without blindly trusting detection.

Visibility (C) falls out for free: the central store IS the queryable surface. The
distributed docs remain source of truth for *content*; the store owns *question state
+ where it propagates*.

## Existing machinery to reuse (from repo scan 2026-06-22)

- **`/brana:log`** — append-only `event-log.md`, free-text `#tags`. A `#question`
  / `#answers` tag slots in with no schema change. Single shared storage surface.
- **`/brana:reconcile`** — already has `consistency` / `security` / `propagation`
  / `knowledge` scopes; the `knowledge` scope already reads `event-log.md`. A
  questions sweep is a natural new scope step (or fold into propagation).
- **`/close` Step 8b PROPAGATE** — already audits "promise-fulfillment debt." Open
  questions are a sibling of promise-debt; close-time is the natural nudge point.
- **~15+ docs/ADRs** already carry ad-hoc `## Open Questions` sections — a
  migration surface, and proof the distributed-marker model is already the de facto reality.

## Scope decision (2026-06-22): proving slice of the graph epic

This feature is **slice 1** of the Universal Doc Graph (`docs/ideas/universal-doc-graph.md`,
t-2223). It stays small but uses **structured targets from day 1** — so the dead-target
check works and the format seeds the future graph. The graph richness (full ontology,
auto-detection, partial-state machine across many edge types) lives in t-2223, not here.

Resolved:
- **Store** — central, per-project, project-agnostic: an **attention register**, not
  just questions. Each record: `id`, `kind` (question | follow-up | owed-answer |
  feedback | …), text, raised date, `status`, `targets[]` (each `{path, anchor,
  status: pending|reconciled}`). "Question" is the primary kind; the register is the
  one place for anything that needs attention / follow-up / an answer / feedback.
- **Reconciliation** — no new engine; the existing `/brana:reconcile` edit-on-approval
  loop, seeded by `(item, answer, target)`. LLM judgment edit per target, human-confirmed.
- **Triggers** — Push (`/brana:log #question` capture, `#answers <id>` resolve) +
  Pull (`/reconcile` sweep that **enumerates** open items, does NOT auto-detect
  answers — judged intractable) + Close backstop (`/close` Step 8b surfaces open ones).

Resolved 2026-06-22:
- **Capture (decided):** all three, AND **Claude proactively files items as they arise
  during work** — not only on request. Explicit `/brana:log #question` for known items
  + Claude auto-file mid-work + `/close` backstop sweeping for unfiled ones. Capture is
  liberal/inescapable so the store never sits empty (the #1 risk).
- **On resolve (decided):** **archive** — resolved items move to a resolved section,
  not deleted. Keeps the provenance trail (what we didn't know + how we resolved it,
  like errata) while keeping the active list clean.
- **Noise control:** liberal capture is made safe by archive-on-resolve + the
  close-time review that curates the *active* list. Liberal-in, triaged-at-close.

## Shape summary

- **Problem:** Resolved answers land in chat, never the docs — docs go stale and dead
  questions pollute context; we rely on tidiness that doesn't scale.
- **Solution:** A per-project **attention register** + the existing `/brana:reconcile`
  loop, so answers get traversed back into the docs/code they block and the item is retired.
- **Audience:** Claude + the operator, on every project (project-agnostic).
- **Scale:** M — a store schema + three skill touch-points (log, close, reconcile).
  Deliberately the *small* slice; the graph epic (t-2223) is separate.
- **Success metric:** open items get reconciled into docs (not chat) without relying on
  memory; near-zero "stale doc because the answer was only ever said in chat" incidents.

## Engineering disciplines

- **DDD (ADR):** attention-register schema (`id`, `kind`, `status`, `targets[]`) +
  capture/reconcile/archive behaviour + how the record format relates to / seeds the
  t-2223 graph. Decision precedes implementation.
- **TDD:** test the *bookkeeping*, not the judgment — capture appends a valid record;
  sweep enumerates only open items; dead-target check flags moved/renamed files; resolve
  moves the record to the archived section; reconcile is seeded with the right targets.
  (The LLM per-target edit is human-confirmed, not unit-tested.)
- **SDD:** update the specs for `log`, `close` (Step 8b), and `reconcile` to document the
  new behaviour; update `docs/reference/` for those skills.
- **Docs:** user-guide entry for the attention register + `#question`/`#answers`; update
  the commands index. Per `/brana:docs` feature strategy.

## Second-order effects

- **Claude proactively files items** → register fills → *surprise:* it creates a
  **visible ledger of intellectual debt** that didn't exist before. Upside: nothing is
  silently dropped. Watch-out: a growing wall of open items can read as overwhelming —
  the archive + close-time triage are what keep it a tool, not a guilt pile.
- **Reconcile is seeded by structured targets** → the `targets[]` data accumulates →
  *surprise/opportunity:* it becomes an **empirical map of which docs are co-affected**,
  i.e. the real fan-out structure. **This slice doesn't just prove the t-2223 format — it
  measures whether the graph epic is justified at all.** t-2222 generates t-2223's
  go/no-go evidence.

## Risks

- **CRITICAL — empty store.** If capturing a question is heavier than typing prose,
  it never happens. Mitigation: capture rides on commands that already run
  (`/brana:log`, `/close`); make structured capture the path of least resistance.
- **HIGH — target staleness.** Renamed/moved/deleted target files leave records
  pending forever and silently wrong (the `spec-graph.json` disease). Mitigation:
  dead-target check on every sweep — flag, don't trust.
- **MEDIUM — auto-detection temptation.** Resist building answer-detection; the sweep
  enumerates, the human confirms. Detection is noise-or-misses.

## Challenge — six hats (2026-06-22)

**⬜ Facts:** Touches 3 skills (log/close/reconcile) + new schema + sweep + dead-target
check + project-agnostic + proactive-filing behaviour. Brana ALREADY has a reminders
system (`brana remind`, ~1000+ pending). Archive precedent: `event-log.md` trims at 90d.

**⚫ Black (risks):**
- **HIGH — register overlaps `brana remind`.** The broadened kinds (follow-up, feedback)
  often have NO doc target — resolving them changes no doc. Those are *reminders*, and
  brana already has that tool. The register risks reinventing it and splitting attention
  across two systems.
- **HIGH — effort is L, not M.** Scope crept since the M estimate: broadened kinds +
  proactive filing + archive + dead-target check + project-agnostic. The reconcile-seeding
  half (map item→targets→edit-loop, rename detection) is the genuinely hard, risky part.
- **MEDIUM — proactive auto-filing is a soft instruction, not a mechanism.** "Claude
  should file as things arise" decays exactly like "Claude should close" — it relies on
  the discipline the feature exists to replace. If it's load-bearing, capture fails silently.
- **MEDIUM — archive unbounded growth** only bites if it sits in the active scan path.

**💛 Yellow (what holds):** Core mechanism (store + targets + reconcile loop + archive)
is unchanged and sound. The `kind` field is one column, not new architecture. Provenance
trail is genuinely valuable. Seeds t-2223's go/no-go data.

**💚 Green (fixes):**
- **Constrain the register to target-bearing items.** Rule: every item MUST have ≥1
  target (a doc/code location resolving it changes). No target → it's a `brana remind`,
  not a register item. This dissolves the overlap AND auto-throttles auto-filing noise
  (Claude only files things with real doc impact).
- **Make `/close` the RELIABLE capture path** (deterministic sweep that always runs);
  Claude-proactive-filing is best-effort bonus, never load-bearing.
- **Split storage:** active items in the scanned store; resolved → separate archive file
  OUT of the hot path. Growth becomes harmless → **no TTL needed.**
- **Split the task:** capture+store+close-surface = M slice ("never lose a question");
  reconcile-seeding + dead-target check = separate slice ("auto-propagate", the hard half).

**🎩 Blue (verdict):** Sound design, ship it — with: (1) target-required constraint
(kills the remind overlap), (2) re-estimate to L and split capture-slice from
reconcile-slice, (3) archive in a separate file, no TTL, (4) demote proactive-filing to
best-effort behind the deterministic close sweep.

**Decided 2026-06-22 (all four fixes accepted):**
- **Register scope = target-required (Option A).** An item enters the register only if
  resolving it changes a doc/code file (≥1 target). Rule: *"when resolved, do I edit a
  document? yes → register, no → `brana remind`."* Keeps `brana remind` for target-less
  to-dos; no overlap; auto-throttles proactive-filing noise.
- **Unified attention system (absorb `remind`) = deferred to the t-2223 epic**, whose
  "everything is nodes" thesis is exactly that. Not this slice.
- Archive → separate file, no TTL. `/close` sweep = reliable capture; Claude proactive
  filing = best-effort bonus. Effort → **L**, split: capture-slice (M) + reconcile-slice.
